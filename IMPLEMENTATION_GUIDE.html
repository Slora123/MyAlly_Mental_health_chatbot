Detailed Build Guide

RAG + Empathy + Safety

Three Dataset Design

Empathetic Student Mental-Health Chatbot From Scratch

This guide explains how to design and implement the app from scratch using the three datasets already present in the project: dataset reddit, empatheticdialogues, and mhqa-main. It is intentionally implementation-oriented but not a full code dump. The goal is to help you make correct design decisions, structure the project cleanly, and build a RAG pipeline that is empathetic, grounded, and safer for student support use cases.

Core principle: do not use one mixed retrieval pool for everything. Treat empathy and factual knowledge as different jobs. Your chatbot should retrieve how to respond from empathy datasets and what factual content may help from MHQA.

1. Product Scope

Before choosing libraries or writing code, define the exact role of the chatbot. This system is best positioned as an empathetic support and psychoeducation assistant for students, not as a therapist, diagnostician, or crisis responder.

What the app should do

  • Understand student distress such as stress, loneliness, confusion, burnout, homesickness, and emotional overwhelm.
  • Reply in an emotionally validating way.
  • Use grounded knowledge when the student asks factual mental-health questions.
  • Offer one or two practical, low-risk next steps.
  • Escalate to safer language when the message looks high-risk.

What the app should not do

  • Diagnose depression, anxiety, trauma, OCD, or any other condition.
  • Recommend medication changes.
  • Pretend to replace campus counseling, therapy, or emergency services.
  • Use raw internet-style replies as-is.
  • Answer a crisis message with a normal friendly chat response.
Important design boundary: MHQA gives grounded mental-health Q/A style knowledge, but it is still not a therapy manual. Use it to support psychoeducation, not to make clinical judgments.

3. System Architecture

The architecture should explicitly separate data preparation, retrieval, generation, and safety checks. A clear separation will save you from mixing concerns later.

RAG architecture at a glance

Yes, this app is designed as a real RAG system. The retrieval architecture has two distinct layers:

  • Empathy RAG layer: retrieves supportive dialogue patterns from Reddit empathy data and EmpatheticDialogues.
  • Knowledge RAG layer: retrieves grounded mental-health knowledge from MHQA.

These two layers are kept separate on purpose. The model first learns how to respond from empathy retrieval and then adds what information may help from knowledge retrieval.

Index-time RAG structure

dataset reddit
    -> clean text
    -> remove level 0 rows
    -> build empathy documents

empatheticdialogues
    -> rebuild dialogue pairs
    -> attach emotion and prompt metadata
    -> build empathy documents

mhqa-main/datasets
    -> normalize question-answer fields
    -> keep valid rows
    -> build knowledge documents

all processed documents
    -> embed with sentence-transformers
    -> store in Chroma
       - student_support_empathy
       - student_support_knowledge

Query-time RAG structure

user message
    -> safety check
    -> intent/router check
    -> retrieve empathy examples
    -> retrieve MHQA knowledge if needed
    -> rerank and filter
    -> assemble prompt
    -> generate response
    -> log retrieved ids and output

5. Understanding the 3 Datasets

The most important design decision is to understand what each dataset is actually good for. They are not interchangeable.

Dataset 1: Reddit empathy annotations

Folder: dataset reddit

Files: emotional-reactions-reddit.csv, interpretations-reddit.csv, explorations-reddit.csv

Observed fields: sp_id, rp_id, seeker_post, response_post, level, rationales

Use for: empathy mechanism examples and supportive response style.

Dataset 2: EmpatheticDialogues

Folder: empatheticdialogues

Files: train.csv, valid.csv, test.csv

Observed fields: conv_id, utterance_idx, context, prompt, speaker_idx, utterance, selfeval, tags

Use for: natural empathetic dialogue flow and emotion-conditioned replies.

Dataset 3: MHQA

Folder: mhqa-main\datasets

Files: mhqa.csv, mhqa-b.csv

Observed fields: topic, type, question, options, correct answer, and validity flags in some files.

Use for: factual mental-health grounding and psychoeducational answers.

Dataset Main contribution Should not be used for
Reddit empathyExamples of emotional reaction, interpretation, and explorationUnfiltered direct answer generation
EmpatheticDialoguesConversation shape, emotion labels, gentle follow-up styleClinical facts or diagnosis-like claims
MHQAMental-health topic grounding, Q/A retrievalTherapeutic empathy or crisis counseling

6. Dataset Processing Pipeline

This is the most important engineering phase. If the data processing is weak, retrieval quality will be weak even if the model is strong.

Step 6.1: Audit each CSV before transforming

Start by creating a profiling script for every file. For each CSV, compute row count, column names, null count per column, duplicate count, label distribution, and average text length. This profiling step is not optional. You need it to write correct filters for each dataset. 

# sketch only
df = pd.read_csv(path)
print(df.columns)
print(df.shape)
print(df.isna().sum())
print(df.duplicated().sum())

Step 6.2: Normalize text consistently

  • strip leading and trailing whitespace
  • collapse repeated internal whitespace
  • normalize line breaks
  • replace dataset placeholders such as _comma_ in EmpatheticDialogues
  • drop obviously broken or empty rows
  • keep original wording mostly intact because tone matters for empathy retrieval
EmpatheticDialogues contains placeholders such as _comma_. Replace them before indexing, or the model may copy those unnatural tokens into replies.

Step 6.3: Process Reddit empathy data

Treat the Reddit files as empathy-annotation datasets, not as directly safe training conversations. Their value is in the labeled relationship between a support-seeking post and a response.

Keep these fields

  • seeker_post as the distress message
  • response_post as the example response
  • level as empathy quality
  • rationales as evidence of what makes the response empathetic

Apply these filters

  • drop rows with empty seeker_post or response_post
  • drop rows where level = 0
  • deduplicate identical seeker-response pairs
  • manually inspect samples from level = 1 to ensure the tone is still acceptable

Build one empathy document per row

{
  "id": "reddit-emotional_reaction-123",
  "text": "Support-seeking message: ...\nEmpathetic response: ...\nEmpathy mechanism: emotional_reaction\nEmpathy evidence: ...",
  "metadata": {
    "source": "reddit",
    "mechanism": "emotional_reaction",
    "quality_score": 2
  }
}

Step 6.4: Process EmpatheticDialogues

Do not index each row independently. Reconstruct adjacent turns so that the indexed item mirrors the runtime task: a student-like message followed by an empathetic reply.

  • sort by conv_id and utterance_idx
  • group by conv_id
  • pair each assistant turn with the immediately preceding user turn
  • use context as emotion metadata
  • use prompt as situation metadata
{
  "id": "empatheticdialogues-train-hit:0_conv:1-2",
  "text": "Emotion label: sentimental\nSituation prompt: ...\nSpeaker message: ...\nEmpathetic reply: ...",
  "metadata": {
    "source": "empatheticdialogues",
    "emotion": "sentimental",
    "split": "train",
    "quality_score": 2
  }
}

Step 6.5: Process MHQA

MHQA should be processed into a pure knowledge index. It should not be treated as a conversational style dataset.

  • normalize correct_answer and correct_option into one output field
  • keep topic and type as metadata
  • if valid_question exists, keep only valid rows
  • drop rows missing the core question or answer field
{
  "id": "mhqa-gold-125",
  "text": "Topic: Depression\nQuestion type: Preventive\nQuestion: ...\nCorrect answer: ...\nOptions: ...",
  "metadata": {
    "source": "mhqa",
    "topic": "Depression",
    "question_type": "Preventive",
    "quality_score": 2
  }
}
Do not mix MHQA rows into the same collection as empathy examples. If you do, a student sharing emotional pain may retrieve an exam-style multiple-choice record instead of a warm support pattern.

7. Unified Document Schema

Even though the sources differ, your indexing pipeline should produce records in a consistent schema. This makes retrieval, logging, debugging, and evaluation easier. 

{
  "id": "unique-string-id",
  "collection": "empathy | knowledge",
  "text": "document content used for embedding and retrieval",
  "metadata": {
    "source": "reddit | empatheticdialogues | mhqa",
    "subset": "optional",
    "quality_score": 0,
    "topic": "optional",
    "emotion": "optional",
    "mechanism": "optional",
    "question_type": "optional",
    "path": "relative path to source file"
  }
}
Field Purpose
sourceShows where the document came from.
quality_scoreAllows reranking and filtering low-value empathy examples.
mechanismDistinguishes emotional reaction, interpretation, and exploration.
emotionHelps match similar emotional situations from EmpatheticDialogues.
topicSupports MHQA topic routing such as depression, anxiety, trauma, or OCD.
question_typeUseful for analytics and fine-grained routing.

For these datasets, one row or one dialogue pair should usually become one indexable unit. You do not need heavy token-based chunking because the source items are already short and semantically compact.

8. Embeddings and Vector Indexing

Once the records are standardized, the next stage is embedding and indexing. This is where the app becomes a real RAG system rather than a plain chatbot.

Recommended embedding approach

Use sentence-transformers for dense semantic embeddings. The library is built specifically for semantic search, embeddings, and retrieval-style applications, which is why it fits this project well. 

from chromadb.utils import embedding_functions

embedding_fn = embedding_functions.SentenceTransformerEmbeddingFunction(
    model_name="all-MiniLM-L6-v2",
    device="cpu"  # or cuda if available
)

Vector database design

Create two Chroma collections:

  • student_support_empathy
  • student_support_knowledge
client = chromadb.PersistentClient(path="./chroma_db")

empathy = client.get_or_create_collection(
    name="student_support_empathy",
    embedding_function=embedding_fn
)

knowledge = client.get_or_create_collection(
    name="student_support_knowledge",
    embedding_function=embedding_fn
)
Chroma documentation describes PersistentClient as suitable for local development and testing, and recommends server-backed Chroma for production use.

Index build process

  1. load processed empathy and knowledge records
  2. batch them into manageable upsert sizes, usually 100 to 500
  3. store id, text, and metadata
  4. rebuild the collection whenever your processing rules change
  5. save the build date and processing version so results are traceable

9. Retrieval Strategy

Retrieval should not be a single fixed query against one pool. Build a small routing layer first, then query the right collection or both collections.

Step 9.1: Add a lightweight message router

  • support-only: emotional distress, venting, loneliness, self-doubt, academic stress
  • knowledge-seeking: asks factual questions about symptoms, coping, therapy, or disorders
  • mixed: needs both emotional validation and factual clarification
  • high-risk: self-harm, suicide, immediate danger, abuse, or inability to stay safe

Step 9.2: Query empathy collection

Always query empathy for support-only and mixed messages. Retrieve several candidates first, then rerank them.

  • initial top_k = 6
  • final empathy snippets passed to the model: 2 or 3

Step 9.3: Query knowledge collection only when needed

Only query MHQA for knowledge-seeking and mixed messages. This prevents the chatbot from sounding too academic when the user mainly needs comfort.

  • initial top_k = 3
  • optionally filter by topic when obvious topic keywords appear

Step 9.4: Rerank retrieved items

After vector search, apply simple project-specific reranking:

  • prefer higher quality_score for Reddit
  • prefer topical relevance for MHQA
  • prefer source diversity so the model does not overfit to one example
  • drop empathy examples with unacceptable phrasing after a safety review pass
# pseudocode only
retrieved = empathy_collection.query(...)
sorted_items = sort_by(
    semantic_similarity,
    quality_score,
    source_diversity
)

Step 9.5: Keep empathy and knowledge separate in the prompt

Label the retrieved context explicitly as empathy context and knowledge context. Do not merge them into one anonymous text block.

10. Prompt and Generation Design

The prompt is where the whole design comes together. The model must be told how to use the retrieved material, not just shown the material.

System prompt responsibilities

  • define the bot as an empathetic student support assistant
  • forbid diagnosis and overclaiming
  • tell the model to use empathy snippets for style and response pattern
  • tell the model to use knowledge snippets for grounded explanation only when relevant
  • force emotional validation first
  • define a safer response mode for high-risk inputs

Prompt structure

System:
You are an empathetic mental-health support chatbot for students...

User message:
{student_message}

Recent history:
{recent_history}

Retrieved empathy examples:
1. ...
2. ...
3. ...

Retrieved mental-health knowledge:
1. ...
2. ...

Instructions:
- Validate the student's feelings first.
- Do not copy retrieved text directly.
- Use factual knowledge only if it clearly helps.
- Do not diagnose.
- Offer one practical next step.
- End with one gentle follow-up question.

Generation settings

Parameter Starting recommendation Why
temperature0.3 to 0.6Keeps replies stable and less erratic.
max_tokens200 to 320Enough space for warmth without rambling.
top_poptional, moderateUse only if you need additional diversity control.

Model serving choice

If you stay close to the current repo, use Hugging Face InferenceClient first. The official docs support chat-completion style calls with role-based messages, which matches the current app design. 

from huggingface_hub import InferenceClient

client = InferenceClient(model="your-chat-model", token=HF_TOKEN)
response = client.chat_completion(messages=messages)

11. Safety Layer

A mental-health chatbot needs a safety layer outside of the normal RAG flow. This should happen both before generation and after generation.

Pre-generation checks

  • detect high-risk self-harm or suicide language
  • detect immediate danger or inability to stay safe
  • detect requests for diagnosis, medication, or emergency advice

High-risk route design

  • acknowledge distress
  • encourage immediate contact with a trusted person, local emergency services, or local crisis support
  • avoid long analytical advice
  • ask one grounding or next-step question only if appropriate

Post-generation checks

  • does the reply contain a diagnosis claim?
  • does it sound blaming, dismissive, or overly casual?
  • does it give unsafe medical advice?
  • does it overstate certainty?
The app should never say things like “you definitely have depression,” “stop taking your medicine,” or “ignore those thoughts.” Those are system-design failures, not small wording issues.

12. App Design

There are two sensible ways to design the app: a fast prototype path and a production path.

Option A: Prototype with Gradio

  • single Python app
  • gr.ChatInterface for the UI
  • easy local testing
  • best for demo and iteration

Option B: Production app later

  • FastAPI backend for routing and inference
  • React or Next.js frontend for a polished student-facing UI
  • better auth, analytics, and feedback collection

Recommended UI features

  • chat history window
  • clear note that the assistant is supportive but not emergency care
  • feedback controls such as “helpful” or “not helpful”
  • optional groundedness badge for factual answers
  • calm, non-clinical visual design suitable for students

Backend responsibilities

  • load config and API tokens
  • load vector collections
  • receive user message and history
  • run routing and safety checks
  • perform retrieval
  • assemble prompt
  • generate and log the answer

13. Conversation Memory

Do not store full runtime conversation history in the same vector collections as your datasets. Runtime memory and offline knowledge are different things.

  • keep the last 3 to 5 turns directly in the prompt
  • log each turn to JSONL or a small database
  • do not index user conversations into empathy or MHQA collections by default
  • if long-term memory is later needed, create a separate memory store with stricter privacy rules

14. Evaluation Plan

Evaluation should be planned before deployment. Mental-health support systems can feel good in demos and still fail in important edge cases.

Build a test prompt set

  • exam and deadline stress
  • loneliness and homesickness
  • low mood and lack of motivation
  • knowledge-seeking questions about symptoms or coping
  • high-risk crisis-like prompts for safety routing tests

Score each response on four axes

Axis Question
EmpathyDid the answer make the student feel heard and understood?
GroundednessWhen factual content appeared, was it aligned with retrieved MHQA context?
SafetyDid the answer avoid diagnosis, unsafe advice, or dismissive tone?
UsefulnessDid the answer offer a realistic next step for a student?

Track retrieval quality too

  • did the right empathy mechanism show up?
  • did the retrieved examples look emotionally similar?
  • did MHQA retrieval match the user’s topic?
  • were any low-quality Reddit samples still surfacing?

15. Deployment Roadmap

Start small and keep the deployment path incremental.

Local development

  • keep datasets local
  • build Chroma collections with PersistentClient
  • run the app in Gradio
  • log locally to JSONL

Pre-production

  • freeze the data processing pipeline
  • version the index build
  • expand test prompts
  • add response review and safety testing

Production direction

  • move to server-backed Chroma
  • use a proper backend API layer
  • store logs more securely
  • add access control and feedback capture
  • publish local emergency and counseling resources based on deployment region

16. Suggested Implementation Phases

A phased implementation helps prevent building everything at once and then discovering retrieval quality problems too late.

Phase 1: Data understanding

  • profile all three dataset groups
  • confirm columns and row counts
  • write down filtering rules

Phase 2: Processing pipeline

  • build separate cleaning scripts for empathy and knowledge
  • generate standardized JSONL records
  • sample records manually to inspect quality

Phase 3: Indexing

  • build the two Chroma collections
  • verify counts and random retrieval samples
  • adjust metadata and ranking logic

Phase 4: RAG orchestration

  • add message router
  • add crisis detector
  • add empathy retrieval
  • add optional MHQA retrieval
  • assemble prompts

Phase 5: App integration

  • connect the orchestration layer to Gradio
  • display history cleanly
  • log all requests and retrieved document ids

Phase 6: Evaluation and hardening

  • test on curated prompt sets
  • review unsafe or weak answers
  • tighten prompt rules and filters
  • improve reranking and routing

17. Common Pitfalls

  • Using one mixed collection for everything. This causes empathy and knowledge to interfere with each other.
  • Keeping Reddit level = 0 rows. This teaches poor support behavior.
  • Indexing raw EmpatheticDialogues rows without reconstructing pairs. This weakens conversational retrieval.
  • Treating MHQA as direct therapy advice. It is a knowledge source, not a counseling policy.
  • No safety routing. A normal chat answer to a crisis prompt is not acceptable.
  • Too much history in every prompt. This increases cost and noise without improving support quality.
  • No retrieval logging. If the answer is poor, you need evidence of what context was retrieved.
